1Introduction

"Local Activity App" is a mobile application. It is an application that allows users to see local activities. These activities are different events put on by local businesses. Normal users would open the app and see a page that has a list of these different promotions from businesses. This q creates a convenient method for people, and business to easily arrange local activities that brings revenue to the local economy and provides recreation for the local residents.

1.1Purpose

The purpose of "Local Activity App" is to provide the users two way experience by allowing them to view and choose to participate in current activities or create and publish their own activity. By choosing to participate in a current activity the user can send a participation request, which must be approved by the activity owner. On the other hand, if the user choose to create their own activity, the user has the option to set that activity as a date or determine a group participation size. Finally, the users can have access on a list of promotion offered by local business and create their activity.

Additionally the business will pay for having their promotions listed in the "Local Activity App". This document will propose the different requirements identified in order to make the "Local Activity App" a functional application.

1.2Scope – CONOP, Use cases

The Local Activity App is an application for people to post activities/dates and for people to find and join those activities/dates. Businesses can also pay for a select spot on the application for others to view their activities.

The application will allow everyday users to:

  • View activities posted
  • Join an activity
  • Chat with those in the activity to establish details
  • Post an activity
  • Post a date
  • Join a date
  • The user that posted the activity/date would have control of it

The application will allow business to:

  • Post activities that feature their businesses
  • Pay for select spots of where their activity will appear

This application will help people meet new friends and do fun activities together. It will also allow those users to post and go on dates. Businesses will have the ability to promote their deals and activities for a price. This will create revenue from the application.

1.3Product Overview

This application is a social app that will provide opportunities to meet new people while doing fun activities. Activities will be able to be created by businesses and individuals in order to facilitate these opportunities. Funding for the app will come in the form of businesses bidding for priority listings for their sponsored activities. The application will provide a simple experience so that focus can be placed on the activities and social connections of participants themselves

1.3.1Product Perspective

The application is like Mutual and Facebook in that it allows for the formation of social connections and event planning.

Third party business software that companies wish to link to, and social media apps, specifically Twitter, Facebook, and Instagram

1.3.1.5Communication Interfaces

The application will enable communication using group chats for specific activities and a direct line to a support team should help be needed. These will all be available through the user interface.

1.3.1.6Memory

Each activity will remain open and viewable from its creation date until the completion of the activity based on the date and time entered. After an activity has been closed a history will be able to be reviewed for up to a month, after which access to activity history will no longer be possible. For legal purposes, a record of every activity and messages over chat will be kept for 3 years.

1.3.1.7Operations

Operations will include push notifications regarding activities, events, messages, etc. Participants will voluntarily join activities and events but then be automatically removed at the conclusion of the event. Regular automatic synchronizations with the server will be made to update memory and records.

1.3.1.8Site Adaptation Requirement

All chat and activity data will be formatted for transport. All chat and activity history data over 5,000 and over 1 month will be deleted starting with oldest first.

3.5 Logical database requirements

Backend and database components

3.5.1User Table

The user table shall store all information about a given user and shall be referred to by other tables that involve users.

3.5.1.1The user table shall contain a first name column.

3.5.1.2The user table shall contain a last name column.

3.5.1.3The user table shall contain the user ID column as the primary key.

3.5.1.4The user table shall contain a role column, indicating whether this user is a typical user or business user.

3.5.1.5The user table shall contain the user's hashed password.

3.5.1.6The user table shall contain the user's Email address (for registration and password retrieving).

3.5.1.7The user table shall contain a rating column, being reported could harm the user’s rating.

3.5.1.8The user ID shall have a foreign key relation to the event table, organizer column.

3.5.1.9The user ID shall have a foreign key relation to the participant table, user ID column.

3.5.1.10The user ID shall have a foreign key relation to the Friendship table, user ID column.

3.5.2Event Table

The event table shall store all information about events and shall be referred to by other tables involving events.

3.5.2.1The event table shall contain the event ID as the primary key.

3.5.2.2The event table shall contain a date column.

3.5.2.3The event table shall contain a time column.

3.5.2.4The event table shall contain a column indicating the maximum participants allowed in the event.

3.5.2.5The event table shall contain an event type column.

3.5.2.6The event table shall contain an event location column.

3.5.2.7The event table shall contain a description column.

3.5.2.8The event table shall contain an organizer column, which should have a foreign key relation to user table, user ID column.

3.5.2.9The event ID column shall have a foreign key relation to the participant table, event ID column.

3.5.3Event-Participant Table

The event-participant table shall represent a pairing of a user ID and an event ID to show that the specified user is attending the specified event.

3.5.3.1The event-participant table shall contain an event-participant ID as the primary key.

3.5.3.2The participant table shall contain a user ID column, which shall have a foreign key relation to user table, user ID column.

3.5.3.3The participant table shall contain an event id column, which shall have a foreign key relation to event table, event ID column.

3.5.4Friendship Table

3.5.4.1The friendship table shall contain a primary key consisting of the user1 and user 2 IDs (user1 ID, user2 ID).

3.5.4.2The friendship table shall contain a user1 ID column, which shall have a foreign key relation to user table, user ID column.

3.5.4.3The friendship table shall contain a user2 ID column, which shall have a foreign key relation to user table, user ID column.
