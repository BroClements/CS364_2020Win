1Introduction

Uvents is a mobile application that brings the community together by allowing personal users to create events, which can be set as a group event or a date event; commercial users to list their promotions, which can be seen by the personal users and added to their events; and admin users to manager event policies.

1.1Purpose

The purpose of Uvents is to provide personal users two way experience by allowing them choose to participate in event created by other users and publish their own events. Additionally, the users can have access on a list of promotion offered by local business and add those promotions to their events. Uvents will receive revenue by having commercial users paying to have their promotions listed in the application. Finally, Uvents provides a convenient way for people to easily arrange local events that brings revenue to the local economy and provides recreation for the local residents.

1.2Scope

Uvents is an application for people to post activities/dates and for people to find and join those activities/dates. Businesses can also pay for a select spot on the application for consumer users to view their activities.

The application will allow consumer users to:

  • View activities posted
  • Join an event
  • Chat with those in the activity to establish details
  • Post a group event
  • Post a one on one date with the opposite sex
  • Post a group date for 4-6 couples
  • Join a date
  • The consumer user that posted the event/date would have control of said event

The application will allow commercial users to:

  • Post activities that feature their businesses
  • Pay for select spots of where their activity will appear

Uvents will help people meet new friends while experiencing an event together. Uvents will also allow those consumer users to post a date or sign up for a date. Businesses will have the ability to promote their events for a price. This feature will create revenue from the application.

1.3Product Overview

Uvents is a social app that will provide opportunities to meet new people while doing fun activities. Events will be able to be created by businesses and individuals in order to facilitate these opportunities. Businesses can create events which can be joined. Funding for the app will come in the form of businesses bidding for priority listings for their sponsored activities. The application will provide a simple experience so that focus can be placed on the activities and social connections of participants themselves.

1.3.1Product Perspective

The application is like Mutual and Facebook in that it allows for the formation of social connections and event planning.

1.3.3 User Characteristics

Uvents will have 3 types of users: consumer, commercial, and admin. The following describes how each of thier needs will be met.

1.3.3.1 Consumer

The Consumers perceive the application as a way to be notified of events in their relative location. The application will be a platform where the Consumer receive notifications of the

1.3.3.2Commercial

From the perspective of the Commercial user this app will be a platform where they can advertise events, activities, and products that the businesses offer. This will help them to attract customers and increase visibility with the public. As an example, Fat Cats could purchase an event listing slot in the app and advertise their movie theater, then consumers who use the app to find ideas for group events and dates would see that event listing.

1.3.3.3Admin

The administrator perceives the app as a medium for businesses to attract users with promotions and events. Admin has certain duties and obligations which must be fulfilled. To fulfill these duties, they would like the app to be easy to update and manipulate. The Admin seeks to make the app profitable by attracting and retaining users and businesses. To retain users, it is crucial that the administrator can fulfil the needs of both users and businesses.

3SPECIFICATION

The Specification explains the detailed requirements for the Uvents application. The Specification also covers supporting information found from using the elicitation techniques.

3.1 External interfaces

3.1.1 Amazon AWS Database

An Amazon AWS Database service shall be used to store user data from the application.

  • 3.1.1.1 The database shall store consumer user accounts [2.5].
  • 3.1.1.2 The database shall store commercial user accounts [2.5].
  • 3.1.1.3 The database shall store admin user accounts [2.5].
  • 3.1.1.4 The database shall store events.
  • 3.1.1.5 The database shall store group events.
  • 3.1.1.6 The database shall store monthly bidding information for commercial user accounts.
  • 3.1.1.7 The database shall store information about connected consumer user social media accounts [3.1.3-3.1.5].
  • 3.1.1.8 Upon request commercial users shall receive a report of their bidding history [2.6].

3.1.2 Google Maps API/GPS

Google Maps API will be used to map the available local events and group events posted by commercial users and consumer users.

  • 3.1.2.1 The map feature shall allow consumer users to view local events. [2.7 (9), Brainstorming 8, Domain Analysis 1]
  • 3.1.2.2 The map feature shall allow consumer users to view group events. [2.7 (9), Brainstorming 8, Domain Analysis 1]
  • 3.1.2.3 The map feature shall allow consumer users to add event locations. [2.7 (9), Brainstorming 8, Domain Analysis 1]
  • 3.1.2.4 The map feature shall allow commercial users to add group event locations. [2.7 (9), Brainstorming 8, Domain Analysis 1]
  • 3.1.2.5 The map feature shall allow consumer users to view their distance from local event locations.
  • 3.1.2.6 The map feature shall allow consumer users to view their distance from local group event locations.
  • 3.1.2.7 The map feature shall allow consumer users to view directions to local event locations
  • 3.1.2.8 The map feature shall allow consumer users to view directions to local group event locations.

3.5Logical Database Requirements

The database shall have all tables necessary to store information required by the application. These tables will consist of a User table, Event table, Event-Participant table, and Friendship table.

3.5.1User Table

The user table shall store all information about a given user and shall be referred to by other tables that involve users. (PF 1)

3.5.1.1The user table shall contain the user ID column as the primary key.

3.5.1.2The user table shall contain a first name column.

3.5.1.3The user table shall contain a last name column

3.5.1.4The user table shall contain a role column.

LD 1.4.1This will indicate whether this user is a consumer user or commercial user.

3.5.1.5The user table shall contain a username (PF 1.9.2)

3.5.1.6The user table shall contain an optional column for a user’s Facebook user ID for users who have logged in via Facebook. (PF 1.8.2) This is separate from the user’s internal user ID (3.5.1.1).

3.5.1.7The user table shall contain the user's hashed password. (PF 1.1.2)

LD 1.4.1Hashed passwords will be excluded from third-party login accounts.

3.5.1.8The user table shall contain the user's Email address (for registration and password retrieving).

3.5.1.9The user table shall contain the user's phone number for registration (for contacting and password retrieving).

3.5.1.10The user table shall contain a rating column.

LD 1.4.1The user’s rating is affected by negative reports given by other users as well as the number of events the user has attended/hosted (3.5.1.11, 3.5.1.12, 3.5.5).

3.5.1.11The user table shall contain a column indicating the number of events the user has attended.

3.5.1.12The user table shall contain a column indicating the number of events the user has hosted.

3.5.1.13The user table shall contain a gender column.

LD 1.4.1The gender column will indicate “male,” “female,” or “unspecified.”

3.5.2Event Table

The event table shall store all information about events and shall be referred to by other tables involving events.

3.5.2.1The event table shall contain the event ID as the primary key.

3.5.2.2The event table shall contain a start date column.

3.5.2.3The event table shall contain an end date column.

3.5.2.4The event table shall contain a start time column.

3.5.2.5The event table shall contain an end time column.

3.5.2.6The event table shall contain a column indicating the maximum participants allowed in the event. (PF 1.10.1)

3.5.2.7The event table shall contain a column indicating max number of male participants.

3.5.2.8The event table shall contain a column indicating max number of female participants.

3.5.2.9The event table shall contain a column indicating max number of unspecified gender participants.

3.5.2.10The event table shall contain an event origin column.

LD 2.10.1to specify whether the event is a user-created or business-created event.

3.5.2.11The event table shall contain an event location column.

LD 2.11.1Location will be stored as customer-created text.

3.5.2.12The event table shall contain an event location pin column.

LD 2.12.1Location pin will be stored as a string representing latitude and longitude.

3.5.2.13The event table shall contain a description column. (PF 11.1.2)

3.5.2.14The event table shall contain an event host column, which should have a foreign key relation to user table, user ID column. (PF 1.10)

3.5.2.15The event table shall contain a promotion points column.

LD 2.15.1Promotion points will determine a business-created event’s rank when displayed by other business-created events.

3.5.2.16The event table shall contain a column for a price description of an event.

3.5.3Event-Participant Table

The event-participant table shall represent a pairing of a user ID and an event ID to show that the specified user is attending the specified event. This will enable this information to be viewed with the event, and this will also enable the use of event-specific group messages (PF 1.8.2, 1.10.2)

3.5.3.1The event-participant table shall contain an event-participant ID as the primary key.

3.5.3.2The participant table shall contain a user ID column, which shall have a foreign key relation to user table, user ID column.

3.5.3.3The participant table shall contain an event id column, which shall have a foreign key relation to event table, event ID column.

3.5.3.4The participant table shall contain an approval status column.

LD 3.4.1Approval status will indicate whether the participant has a status of “invited,” “requested,” or “approved.” (PF 1.11.4, 1.11.5, 1.12.1)

3.5.4Friendship Table

The friendship table shall show a contact relationship between two users so that these users can interact differently from how they interact with non-friend users. (PF 1.8.1)

3.5.4.1The friendship table shall contain a friendship ID as the primary key.

3.5.4.2The friendship table shall contain a user1 ID column, which shall have a foreign key relation to user table, user ID column.

3.5.4.3The friendship table shall contain a user2 ID column, which shall have a foreign key relation to user table, user ID column.

3.5.5Report Table

The report table shall show a relationship between a reported user, a reporting user, a report type, and an associated event (if applicable).

3.5.5.1The report table shall have a required reporting user column, which shall have a foreign key relation to user table, user ID column.

3.5.5.2The report table shall have a required reported user column, which shall have a foreign key relation to user table, user ID column.

3.5.5.3The report table shall have a required report type column.

LD 5.3.1Approval status will indicate whether the participant has a status of “invited,” “requested,” or “approved.” (PF 1.11.4, 1.11.5, 1.12.1)

3.5.5.4The report table shall have an event column, which shall have a foreign key relation to event table, event ID column.

LD 5.3.1The event will allow checking to make sure a no-show user’s rating isn’t affected multiple times for missing one event.

3.5.6Conversation Table

The conversation table shall store all message conversations to be referred to when knowing what consumer users should be given access to the conversation and what messages are assigned to it.

3.5.5.1The conversation table shall include a conversation ID as the primary key.

3.5.7User-Conversation Table

The user-conversation table shall indicate a relationship between a user and a conversation they have access to.

3.5.7.1The user-conversation table shall have a foreign key relationship to the user table, user ID column.

3.5.7.2The user-conversation table shall have a foreign key relationship to the conversation table, conversation ID column.

3.5.8User-Conversation Table

The message table shall store individual messages, including their content, the time they were sent, the originator, and the conversation they belong to.

3.5.8.1The message table shall have a message ID as the primary key.

3.5.8.2The message table shall have a originator column with a foreign key relationship to user table, user ID column.

3.5.8.3The message table shall have a conversation column with a foreign key relationship to the conversation table, conversation ID column.

3.5.8.4The message table shall have a content column containing the body of the message.

3.5.8.5The message table shall have a time column referring to the date and time the message was created.

3.5.9Business Table

The business table shall store all the information needed to represent a given business and will be linked to a commercial user’s user ID for reference.

3.5.9.1The business table shall have a business ID column as the primary key.

3.5.9.2The business table shall have an owner column with a foreign key relationship to user table, user ID column.

3.5.9.3The business table shall have a description column.

3.5.9.4The business table shall have an address column.

3.5.9.5The business table shall include a phone number column.

3.5.9.3The business table shall include an email column.

3.5.9.4The business table shall include a category column.

3.5.9.5The business table shall include a rating column.

3.5.10Payment Table

The payment table shall store information about credit/debit cards used by commercial users to promote their events.

3.5.10.1The payment table shall include a foreign key relationship to the business table, business ID column.

LD 10.1.1The event will allow checking to make sure a no-show user’s rating isn’t affected multiple times for missing one event.

3.5.10.2The payment table shall include a column for the name on the card.

3.5.10.3The payment table shall include a column for the card number.

3.5.10.4The payment table shall include a column for the expiration date on the card.

3.5.10.5The payment table shall include a column for the billing address to be used for payments.

3.5.11Tag Table

The tag table shall assign a given tag to a business.

3.5.11.1The tag table shall include a tag ID column as the primary key.

3.5.11.2The tag table shall have a foreign key relationship to business table, business ID column.

3.5.11.3The tag table shall have a tag column storing the name of a tag (e.g. “Sports” or “Art”).

3.5.12Rating Table

The rating table shall include information about a rating given to a business by a consumer user.

3.5.12.1The rating table shall include a rating ID column as the primary key.

3.5.12.2The rating table shall include a business column with a foreign key relationship to the business table, business ID column.

3.5.12.3The rating table shall include an originator column with a foreign key relationship to the user table, user ID column.

3.5.12.4The rating table shall include an originator column with a foreign key relationship to the user table, user ID column.
