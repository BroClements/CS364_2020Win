1Introduction

1.1Purpose

This should reference and describe the product, not the company, customer, or the SRS document.

1.2Scope – CONOP, Use cases

You are putting boundary's of what the product is and in rare case what it is not.

You should introduce the Concept of Operations and Use Cases

1.3Product Overview

Perspective –

CONOP, ROI, Architecture, Stakeholders, Business Scope, Legal Functions – Overall characteristics & features, non-formal, mapped from elicitation data User Characteristics – who is using the application, user stories, workflows, etc. Limitations - what it will not do, storage & performance constrains, etc.

1.4Definitions – titles, terms.

If you give a name to anything it should be defined here.

3SPECIFICATION

3.1 External interfaces

You reference the external interface, here you are specifying specifics about that interface.

Much like elicitation data, you are collecting and reporting information that will be used to create requirements.

You can include requirements in this section

3.2 Functions

From the features or product functions, you will be deriving components of the system. These components will be named and are sub-system elements of the product. These components can be categorized as Front-End, Middleware, Backend, and Database. (Don’t use the category names, just mentioned).

3.3 Usability Requirements

More than HCI Usability, includes Interfaces and fulfilling the users needs

3.4 Performance requirements

Magic numbers are ok here as explanations to requirements

3.5 Logical database requirements

Backend and database components

3.6 Design constraints

System and Component requirements dealing with End to End (Full-Stack)

3.7 Software system attributes

Requirements that don’t fit neatly in a single component.

Kitchen sink, Summary of the Elicitation data

4VERIFICATION

4.1Verification uses similar specific wording.

i.e.: The system shall allow users to withdraw cash

i.e.: Verify that the system shall allow user to withdraw cash

4.2Next how will that happen

Provide a brief description of what needs to be verified.

Then provide additional detail on the information needed to do the verification.

i.e.: Verify that the system shall allow user to withdraw cash

User will enter verification information and request from the system specified amount, if user has sufficient funds, the system will then return to the user cash, and update user account amount.

If there isn't sufficient funds, the system will notify the user, system will not return any cash

Verification section can be a separate section or a sub-section in the same section as the requirements. (Should map 1-to-1 or 1-to-n, avoid n-to-1, don’t use n-to-m)
